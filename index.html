What's happening

Government departments each build their own versions of identical digital components — NI validators, address lookups, identity checks — with no awareness that the others exist.

Today
HMRC, DWP, DVLA and others each maintain their own copy of near-identical logic. Three NI validators. Four address lookups. All drifting apart independently.

Why this matters
A developer at HMRC spends three weeks building an NI number validator. That same month, a developer at DWP builds the same thing. Neither knows the other exists. Both will need to be maintained, patched, and updated separately — forever. This isn't a one-off. It's the default pattern across government digital delivery.

One central GovCore repository holds verified, versioned shared components. Departments keep full ownership of their own satellite repos. They consume GovCore components as dependencies — not copies. Semantic versioning means teams upgrade on their own schedule.

The three core rules
Rule 1: Satellites consume GovCore — they never copy code into their own repo. The moment you copy it, you recreate the divergence problem. Rule 2: Semantic versioning means departments choose when to upgrade. Nobody is forced to take a breaking change immediately. Rule 3: All pull requests to the GovCore core require two approving reviewers from different departments — preventing any single team from controlling the shared platform.

  • RESTful interfaces mean any language, any stack can consume GovCore

The four mechanisms
  • Pull Requests: Any developer from any department can propose a change. Written description, async review, full audit trail. No meeting required.
  • RFC Process (Request for Comments): Before any major change, a short written proposal is shared for comment. Think planning permission before construction — a record of the decision, not a committee.
  • Component Guilds: Each shared component has a small group of named maintainers drawn from its heaviest users. Named accountability — not "everyone's responsibility."
  • Transparent by Default: All code, all review comments, all RFC decisions are visible to every participating department. This is what makes cross-geography collaboration work at scale.

The access model — explicitly
READ access is open to all government departments by default. Any team can view and consume GovCore components without approval. WRITE access — proposing changes — requires guild membership, granted per-component by existing guild members. ADMIN access — merging to main, approving releases — is limited to two named maintainers per component, reviewed annually.
  • Review SLA: Maintainers commit to reviewing PRs within 5 working days — with named escalation if missed. Only works if the team is properly staffed.
  • Automated gates: Unit tests, SAST security scanning, and accessibility checks run on every PR before a human sees it.
  • Opt-out: Teams can always build their own — but document the justification within existing spend approval. Friction, not a wall.
  • Honest caveat: This requires senior sponsorship and a funded team. Without those, the SLA breaks and adoption fails.

Why OpenAPI is the key
Without a shared interface standard, interoperability is always a project — two teams negotiating a custom integration. With GovCore's OpenAPI requirement, every shared component is self-documenting and self-describing from day one. HMRC's Java service and DWP's Python service can both consume the same NI validation component without either team needing to know how the other is built. The contract is the interface — and the contract is enforced at the point of contribution.
  • OpenAPI spec is a condition of merging to GovCore — not optional documentation
  • Machine-readable contracts enable auto-generated client libraries in any language
  • Consistent outputs across all consumers — same input always produces same result
  • This is how GOV.UK Pay and GOV.UK Notify already work — GovCore extends the model
